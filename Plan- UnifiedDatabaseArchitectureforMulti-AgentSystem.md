# Plan: Unified Database Architecture for Multi-Agent System

## TL;DR
The system currently has 4 independent SQLite databases (Discovery, Orders, Quotes, Notifications) with no enforced cross-references, plus 4 agents using in-memory mocks. GCP deployment only syncs 1 database. This plan presents 3 options for consolidation, with a recommended approach (Option B: Unified DB) that merges all tables into a single `sales_agent.db`, adds a `customer_master` table as the central entity, and modifies the GCS sync to handle it.

## Options

### Option A: Multi-DB with GCS Sync (Minimal Change)
Keep separate databases, extend entrypoint.sh to sync all of them.

**Pros:** Minimal code changes, each agent owns its DB
**Cons:** No FK enforcement across DBs, no unified entity model, 4 separate GCS syncs

**Changes needed:**
- Modify `entrypoint.sh` to sync all 4 .db files
- Add env vars for all DB paths in Cloud Run config
- No schema changes

### Option B: Unified SQLite Database (Recommended)
Merge all tables into a single `SuperAgent/data/sales_agent.db` with a new `customer_master` table as the central entity linking everything.

**Pros:** 
- Single DB = single GCS sync, single WAL, simpler ops
- FK enforcement across all tables (orders → quotes, notifications → orders, etc.)
- Central customer_master connects Discovery accounts to all downstream entities
- Single connection pool, simpler threading model
- Matches TODO #1 and #3 from AGENTS.md

**Cons:**
- Requires refactoring all 4 db utility modules to point to shared DB
- Discovery's pre-existing schema needs migration
- Larger single file (but still small for SQLite — <100MB)

### Option C: Cloud SQL (PostgreSQL)
Replace SQLite entirely with managed Cloud SQL.

**Pros:** True concurrent access, no file sync needed, production-grade
**Cons:** Significant refactor (SQL dialect changes), cost ($50-100/mo), overkill for demo/academic project, adds network latency

## Recommended: Option B — Unified SQLite Database

### Design Decisions (Confirmed)

1. **Two-table lifecycle (B2):** Prospects stay in `accounts` (Discovery's domain). `customer_master` only populated after fulfillment completes. Clean separation of sales pipeline vs. customer records.
2. **Early customer_id generation:** `customer_id` (CUST-YYYYMMDD-XXX) generated at Discovery time, stored in `accounts` as new column, flows through all downstream entities via session state. Single ID links the entire lifecycle.

### Abandonment & Lifecycle Design (Confirmed)

**Problem:** Customers can abandon at any stage, leaving orphaned records. ADK sessions are in-memory — conversation context is lost on disconnect/restart.

**Solution:** TTL + Background Cleanup + Resume Capability

**Entity Status Lifecycles:**
```
quotes:       active ──(30 days)──→ expired     ──(order placed)──→ ordered
carts:        active ──(24 hours)──→ expired    ──(order placed)──→ converted
orders:       draft → pending_payment ──(48h)──→ cancelled   ──(paid)──→ paid ──(7d no fulfillment)──→ escalated ──(fulfilled)──→ fulfilled
payments:     pending ──(15 min)──→ timed_out   ──(auth ok)──→ authorized → completed
fulfillments: scheduled → dispatched → installed → activated
```

**Every transactional table gets:** `status`, `created_at`, `updated_at`, `expires_at`

**cleanup_stale_records():** Runs on server startup + hourly background task (FastAPI). Marks expired/cancelled/escalated records per TTL rules above.

**check_customer_state(customer_id):** New tool for DiscoveryAgent. When a returning customer is identified, queries all tables and returns their pipeline position (active quotes, pending orders, etc.) so the conversation can resume.

### Notification Triggers & Post-Notification DB Updates

**Problem:** Only 3 of 10+ lifecycle events auto-send notifications. Key events (quote confirmation, fulfillment stages, cancellation) produce no customer comms. Notifications don't update source tables — you can't tell from `orders` whether the customer was actually notified.

#### Current State (3 auto-send, 4 manual)

| Event | Trigger | Auto? | Source Agent | Notification Type |
|-------|---------|:-----:|--------------|-------------------|
| Order created | `create_order()` → `sys.modules` cross-call | ✅ Auto | OrderAgent | `ORDER_CONFIRMATION` |
| Payment success | `process_payment()` → `sys.modules` cross-call | ✅ Auto | PaymentAgent | `PAYMENT_SUCCESS` |
| Payment failed | `process_payment()` → `sys.modules` cross-call | ✅ Auto | PaymentAgent | `PAYMENT_FAILED` |
| Installation reminder | User asks SuperAgent | Manual | SuperAgent routing | `INSTALLATION_REMINDER` |
| Service activated | User asks SuperAgent | Manual | SuperAgent routing | `SERVICE_ACTIVATED` |
| Order status update | User asks SuperAgent | Manual | SuperAgent routing | `ORDER_STATUS_UPDATE` |
| Abandoned cart | User asks SuperAgent | Manual | SuperAgent routing | `ABANDONED_CART` |

**Gaps (no notification at all):**
- ❌ Quote generated — tool exists (`send_quote_confirmation()`) but OfferManagement never calls it
- ❌ Installation scheduled — `schedule_installation()` sends no notification
- ❌ Technician dispatched — `dispatch_technician()` sends no notification
- ❌ Installation complete — `complete_installation()` sends no notification
- ❌ Service activated — CAN trigger notification but doesn't auto-send
- ❌ Order cancelled (TTL expiry) — no notification
- ❌ Cart abandoned (TTL expiry) — cleanup marks expired but no notification

| Event | Auto? | Notification Type | Post-Notification DB Updates |
|-------|:-----:|-------------------|------------------------------|

#### New Notification Types to Add
- `QUOTE_CONFIRMATION` — sent by OfferManagement after `generate_quote()`
- `INSTALL_SCHEDULED` — sent by ServiceFulfillmentAgent after `schedule_installation()`
- `INSTALL_DISPATCHED` — sent by ServiceFulfillmentAgent after `dispatch_technician()`
- `INSTALL_COMPLETE` — sent by ServiceFulfillmentAgent after `complete_installation()`
- `ORDER_CANCELLED` — sent by `cleanup_stale_records()` when TTL expires a pending order
- `ABANDONED_CART` — sent by `cleanup_stale_records()` when TTL expires a cart (upgraded from manual-only)

```python
        customer_id=customer_id,

    )
```

### Implementation Steps

#### Phase 1: Create Unified DB Module & Schema (blocks all other phases)

1. Create `SuperAgent/data/` as the canonical DB location
2. Create `SuperAgent/super_agent/utils/database.py` — unified DB manager module
   - Single `get_connection()` function returning a WAL-enabled connection to `sales_agent.db`
   - `init_db()` that creates ALL 16 tables listed above with proper FKs and indexes
   - Thread-safe via `threading.Lock()` (consistent with existing pattern)
   - Path configurable via `SALES_AGENT_DB_PATH` env var, default `SuperAgent/data/sales_agent.db`

## Decisions
- **Option B (Unified SQLite)** recommended over multi-DB sync (Option A) or Cloud SQL (Option C)
- **Two-table lifecycle (B2):** Prospects stay in `accounts` (Discovery's domain). `customer_master` only populated after fulfillment completes.
- **Early customer_id:** Generated at Discovery time (`add_company()`), stored in `accounts.customer_id` column, flows through all downstream tables via ADK session state
- Discovery's legacy tables migrated as-is with new `customer_id` column added to `accounts`
- Single `SALES_AGENT_DB_PATH` env var replaces 4 separate DB path vars
- PaymentAgent and ServiceFulfillmentAgent get real DB persistence (replacing in-memory mocks)
- ServiceabilityAgent and ProductAgent remain in-memory (read-only reference data, no state to persist)

## Scope Boundaries
- **Included:** All 4 existing DBs consolidated, 2 new tables (payments, fulfillments), customer_master (post-sale only), GCP sync update, early customer_id generation, expanded notification triggers (10 auto-send covering full lifecycle), dedup_cache table preserved, post-notification DB state updates
- **Excluded:** Cloud SQL migration, session persistence (ADK InMemorySessionService), ServiceabilityAgent/ProductAgent DB persistence (read-only data), SchedulerAgent for appointment_id generation
